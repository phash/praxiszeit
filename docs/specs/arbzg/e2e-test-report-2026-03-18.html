Anwendungs-Bugs (gefunden & behoben)

Während der Test-Entwicklung wurden zwei echte Bugs in der Anwendung entdeckt:

APP-1: Journal-Navigationslink führt zur falschen URL
Bug UX Behoben

Datei: frontend/src/components/Layout.tsx:352

Beschreibung: Der „Journal"-Link in der mobilen Bottom-Navigationsleiste verwies auf /time-tracking ohne den ?tab=journal-Parameter. Dadurch öffnete ein Klick auf „Journal" immer den „Einträge"-Tab statt den Journal-Tab.

Auswirkung: Mitarbeiter, die auf mobilen Geräten auf „Journal" tippen, landen im falschen Tab und sehen ihre Monatsübersicht nicht.

Vorher:

{ to: '/time-tracking', icon: Clock, label: 'Journal' },

Nachher:

{ to: '/journal', icon: Clock, label: 'Journal' },

Hinweis: Die Route /journal in App.tsx redirected korrekt zu /time-tracking?tab=journal (Replace-Redirect).

APP-2: nginx-Konfliktt zwischen /help-SPA-Route und /help/-Verzeichnis
Bug Infrastruktur Behoben

Datei: frontend/nginx.conf

Beschreibung: Das Build-Artefakt enthält ein Verzeichnis /help/ (mit Markdown-Handbuchdateien). Wenn ein Nutzer direkt /help im Browser aufruft, findet nginx das Verzeichnis /help/, sucht nach /help/index.html (nicht vorhanden) und gibt 403 Forbidden zurück – statt auf den React-Router-SPA-Fallback zurückzufallen.

Auswirkung: Direktes Aufrufen von http://praxiszeit.local/help in einem Browser oder nach einem Hard-Refresh schlägt mit 403 fehl. Der Hilfe-Button in der Sidebar öffnet ein Panel und ist nicht betroffen, aber Lesezeichen und direkte Links sind kaputt.

Fix: nginx.conf – neue explizite Location vor dem SPA-Fallback:

location = /help {
    try_files /index.html =404;
    add_header Cache-Control "no-cache, no-store, must-revalidate";
}

Die location = /help (exakte Übereinstimmung) greift vor location / und serviert immer index.html, damit React Router die Route auflösen kann.

Test-Infrastruktur-Bugs (gefunden & behoben)

Die folgende Tabelle listet alle Korrekturen an den Playwright-Tests. Diese Bugs verursachten falsche Fehlermeldungen, waren aber keine Fehler in der Anwendung selbst (mit Ausnahme wo angegeben).

TEST-1: UTC vs. Lokales Datum in date.helper.ts
Kritisch Behoben

Datei: e2e/helpers/date.helper.ts

Beschreibung: formatDate() verwendete toISOString(), das immer UTC zurückgibt. Um Mitternacht in der Zeitzone Europe/Berlin (UTC+1) ist UTC-Datum einen Tag zurück. Das Backend-Python nutzt date.today() in der Serverzeit (Europe/Berlin). Zeiteinträge für „heute" (UTC = March 17) wurden vom Backend abgelehnt, weil der Server „heute" = March 18 sah.

Betroffene Tests: create/edit/delete time entry, ArbZG §3-Warnung.

// VORHER (fehlerhaft):
return d.toISOString().split('T')[0]; // UTC

// NACHHER (korrekt):
const year  = d.getFullYear();
const month = String(d.getMonth() + 1).padStart(2, '0');
const day   = String(d.getDate()).padStart(2, '0');
return `${year}-${month}-${day}`; // Lokale Zeit

TEST-9: Race Condition bei Änderungsantrag-Tests (Timing)
Timing Verbessert

Datei: e2e/tests/employee/change-requests.spec.ts

Beschreibung: Nach dem Navigieren zu /time-tracking wurde ohne Warten auf das Laden der Einträge direkt nach dem Änderungsantrag-Button gesucht. count() gab 0 zurück, der Loop navigierte in den vorherigen Monat, fand den Eintrag nicht mehr.

// NACHHER: Warten auf networkidle + waitFor vor dem Loop
await employeePage.waitForLoadState('networkidle');
const found = await changeRequestBtn.first()
  .waitFor({ state: 'visible', timeout: 3000 })
  .then(() => true).catch(() => false);
TEST-10: Flaky Login / Profilbild-Upload durch Backend-Rate-Limiting
Infrastruktur Bekannt

Dateien: auth/login.spec.ts, employee/profile.spec.ts

Beschreibung: Das Backend begrenzt Login-API-Aufrufe auf 5/min (slowapi). Bei sequenziellem Ausführen aller 114 Tests in einem Worker werden dieses Limit gelegentlich überschritten. Betroffen: Test-Fixture-Login beim Erstellen des Test-Users für Profilbild-Tests.

Verhalten: Tests schlagen im Erst-Lauf fehl, bestehen im automatischen Retry. Dies ist kein App-Bug, sondern eine Limitation des sequenziellen Test-Setups. Abhilfe: Parallelisierung mit mehreren Workern oder Verlängerung der Timeouts.

Flaky Tests (bestehen im Retry)

Diese Tests schlagen gelegentlich im Erst-Lauf fehl, bestehen aber beim automatischen Retry. Es handelt sich nicht um App-Bugs, sondern um Rate-Limiting-Artefakte beim sequenziellen Test-Betrieb.
TestUrsacheHäufigkeit
auth/login.spec.ts – successful login redirects to dashboard Backend-Rate-Limit: /api/auth/login max. 5/min. Bei großen Test-Suites wird das Limit kurzzeitig überschritten. ~20% der Fälle
employee/profile.spec.ts – Profilbild: Upload gültiges JPEG Test-User-Login im beforeEach schlägt durch Rate-Limit fehl (Timeout 60s). ~15% der Fälle

Empfehlung: In der CI-Pipeline die Tests mit --workers=1 und einem Delay zwischen Tests oder separaten Testsuiten für Auth-Tests ausführen.
