📄 frontend/src/utils/errorMessage.ts

Beschreibung

Math.round((abs - h) * 60) kann den Wert 60 zurückgeben, wenn der Bruchteil nahe 1.0 liegt – z.B. 1.9999h → Ausgabe "1:60" statt "2:00". Dieses Problem tritt bevorzugt bei Dezimalwerten auf, die durch Python's Decimal→float-Serialisierung erzeugt werden und Fließkomma-Artefakte enthalten.

Reproduktion

// Beispiel: Python gibt 1.9999999 zurück
const val = 1.9999999;
const h = Math.floor(val);        // → 1
const m = Math.round((val - h) * 60); // → 60  ← Bug!
// Ausgabe: "1:60" (ungültig)

Fix

Empfehlung: Nach dem Runden prüfen:
if (m === 60) { h++; m = 0; }
Alternativ Math.floor statt Math.round verwenden – beide Änderungen sind einzeilig und haben kein Risiko-Potential.

Betroffenes Gesetz

§3 EntgFG – Entgeltfortzahlung im Krankheitsfall

Beschreibung

Für SICK-Abwesenheiten wird absence.hours bei Erstellung gespeichert. Für Nutzer mit use_daily_schedule=True wird der Wert korrekt server-seitig aus get_daily_target_for_date() berechnet. Für Standard-Wochenstunden-User kommt der Wert jedoch vom API-Aufrufer (Formular-Eingabe) ohne server-seitige Validierung gegen den tatsächlichen Tagessoll.

Rechtliches Risiko

Risiko: Ein Admin, der 0h oder einen falschen Wert eingibt, verletzt §3 EntgFG für den Mitarbeiter (Defizit trotz Krankheit) oder erzeugt falsche Überstunden.

Empfehlung

In absences.py für SICK-Abwesenheiten immer server-seitig get_daily_target_for_date() aufrufen, unabhängig von use_daily_schedule.

Betroffenes Gesetz

§3 EntgFG – Entgeltfortzahlung im Krankheitsfall

Beschreibung

Wenn ein Mitarbeitender eine Stundenreduzierung erhält (z.B. 40h→30h/Woche), bleiben alte SICK-Einträge mit den alten Stundenwerten gespeichert. get_monthly_target verwendet get_weekly_hours_for_date() (historisch korrekt), aber get_monthly_actual liest a.hours (möglicherweise veraltet).

Rechtliche Einschätzung: Rechtlich ist der zum Zeitpunkt der Erkrankung gültige Wert korrekt (§3 EntgFG = "wie wenn gearbeitet"), also ist das Verhalten möglicherweise korrekt – aber es ist nicht dokumentiert und akkzidentell richtig. Dokumentation empfohlen.
